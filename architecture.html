Yufa-MT — System Architecture

Yufa-MT is a Go web app that tracks the Project Yufa Ragnarok Online server: market listings, online players, characters, guilds, MVP kills, WoE rankings, in-game chat, plus a community trading post fed by a Discord bot and parsed by Google Gemini. A single Go binary scrapes upstream pages on a schedule, sniffs game packets with libpcap, persists everything to SQLite (WAL + FTS5), and renders html/template pages over plain net/http.

Go 1.24 · CGO · -tags fts5 SQLite (WAL, FTS5) html/template + HTMX + Alpine + Tailwind 12-factor: .env / env vars Single binary · Debian-slim Docker

1 · High-level system processes, data flows, externals

One cmd/server process hosts everything: HTTP, scrapers, the Discord bot, libpcap chat capture, and the visitor logger. All long-lived workers share a single context.Context cancelled on SIGINT/SIGTERM so shutdown drains in-flight work.

Internal package External service / API Persistent store Network capture Browser / user

Boot sequence (cmd/server/main.go → server.Run)

  1. godotenv.Load() → config.Load() validates env, returns *Config
  2. storage.Open() opens SQLite WAL, runs createTables + createIndexes + dynamic MVP table + idempotent migrations
  3. populateItemDBOnStartup() upserts internal_item_db from data/seed/item_db_*.yml (synchronous)
  4. startBackgroundJobs() launches 7 ticker workers + chat packet capture; startDiscordBot() and startVisitorLogger() begin
  5. http.Server.ListenAndServe(); SIGINT/SIGTERM → ctx cancel → Shutdown(10s) → bgWg.Wait() drains every goroutine

2 · Code layout packages and what each one owns

Everything under internal/ is module-private (Go's internal/ rule), so the public surface is just cmd/server. internal/server remains the fat package — handlers, scrapers, models — while config/i18n/middleware/storage/visitor/itemdb/discord/gemini/rms/xp are small, focused, and (where they matter) unit-tested.

3 · Request lifecycle middleware order, render path

Public traffic flows through the outer Gzip wrap, then either the visitor tracker (public routes) or BasicAuth (admin routes), then the handler. Handlers read SQLite, build a view-model, and dispatch to a cached template. Static files skip auth and gain a long cache header.

① TCP / TLS terminationWhatever fronts the binary (reverse proxy or direct on :8080)
② middleware.GzipNegotiates Accept-Encoding, skips Range; pools *gzip.Writer
③ http.ServeMuxPath-based: /admin/* → admin sub-router · /static/*, /emblems/* → file servers · else → public handler
④ Per-route middlewarePublic: visitorTracker (logs page view to buffered channel). Admin: middleware.BasicAuth(adminUser, adminPass)
⑤ HandlerReads database/sql, calls rms/itemdb/xp/i18n as needed, builds the view-model struct
⑥ RenderrenderTemplate picks the page template (parsed at init alongside navbar + pagination) and writes HTML to the response

5 · Background jobs & ingestion tickers, packet capture, Discord

Each job is a goroutine spun up in startBackgroundJobs with its own time.Ticker. runJobOnTicker exits when the shared context cancels, so SIGINT lets every job stop between iterations rather than mid-write.

runJobOnTicker(ctx, Job{Func, Interval}) ctx cancel on SIGINT/SIGTERM time.Ticker per-job Interval scrape func (Job.Func) per-domain mutex guards upstream HTTP net/http + goquery parse + tx regex on Next.js JSON SQLite WAL writer select case tick GET body ctx.Done() → goroutine returns between iterations (graceful shutdown)

Long-lived workers that don't fit the ticker pattern

Chat capture

startChatPacketCapture opens a single pcap.Handle and runs until ctx is cancelled. Streams TCP payloads → channel decoder → batched insert into chat.

Discord bot

internal/discord.Start opens a discordgo session and registers a MessageCreate handler. Each accepted message goes through injected parse (Gemini) and post (server) callbacks.

Visitor logger

visitor.Logger.Run drains a 1000-slot PageView channel, flushing every 10 s or at 100 rows. The middleware enqueues without blocking.

The upstream site is Next.js, so the data sits inside __next_f.push JSON chunks. Scrapers glue the chunk boundaries back together, then run pre-compiled regexes (charBaseLevelRegex, mvpKillsRegex, …). Per-domain mutexes (marketMutex, characterMutex, …) prevent overlapping runs against the same source.

6 · Storage model SQLite tables, indexes, FTS5

Single database (data/runtime/market_data.db) opened with WAL, a 5s busy timeout, and a 10-connection pool to allow parallel reads alongside one writer. Schema and indexes are idempotent; new columns ship as migrations in applyMigrations (e.g. characters.zeny, guilds.is_active, character_changelog.event_kind).

DomainTablesNotes
Market items, market_events, scrape_history Indexes on (name_of_the_item, is_available), (item_id), (is_available, seller_name), timestamp DESC.
Item DB internal_item_db, rms_item_cache, rms_item_cache_fts FTS5 virtual table mirrors rms_item_cache via triggers (_ai, _ad, _au). Indexed on type, slots, LOWER(name), LOWER(name_pt).
Players player_history Timestamp PK; count + seller_count per minute sample.
Characters characters, character_changelog, v_character_changelog FK to guilds.name. event_kind (drop / level_base / guild_join / …) backfilled on migration; indexed by (event_kind, change_time DESC).
Guilds guilds is_active for soft-archived guilds; emblem_local_path points at data/runtime/emblems/.
MVP kills character_mvp_kills Dynamic schema — one "mvp_<mob_id>" INTEGER column per MVP ID passed to storage.Open. Storage knows nothing about the IDs.
WoE woe_seasons, woe_events, woe_event_rankings Composite PK (event_id, character_name); cascade delete from season → events → rankings.
Trading post trading_posts, trading_post_items One Discord message can produce many items. edit_token_hash lets the original author edit without auth.
Chat chat, chat_activity_log Indexes on (channel, timestamp DESC) and (timestamp DESC). chat_activity_log truncates to minute granularity.
Visitors visitors, page_views SHA-256 hash of X-Forwarded-For/RemoteAddr + User-Agent. Indexes on page_path and (visitor_hash, view_timestamp DESC).
Sample DDL excerpts 
CREATE VIRTUAL TABLE IF NOT EXISTS rms_item_cache_fts USING fts5(
    name, name_pt,
    content='rms_item_cache', content_rowid='item_id'
);

CREATE TRIGGER IF NOT EXISTS rms_item_cache_au AFTER UPDATE ON rms_item_cache BEGIN
    INSERT INTO rms_item_cache_fts(rms_item_cache_fts, rowid, name, name_pt)
        VALUES ('delete', old.item_id, old.name, old.name_pt);
    INSERT INTO rms_item_cache_fts(rowid, name, name_pt)
        VALUES (new.item_id, new.name, new.name_pt);
END;

-- dynamic table built at startup from a list of MVP mob IDs
CREATE TABLE IF NOT EXISTS character_mvp_kills (
    "character_name" TEXT NOT NULL PRIMARY KEY,
    "mvp_1086" INTEGER NOT NULL DEFAULT 0,
    "mvp_1115" INTEGER NOT NULL DEFAULT 0,
    -- … one per MVP …
    FOREIGN KEY(character_name) REFERENCES characters(name)
        ON DELETE CASCADE ON UPDATE CASCADE
);

Build & run

make build      # go build -tags fts5 ./cmd/server
make run        # build, free :8080, open browser
make test       # go test -tags fts5 ./...
make vet
make lint       # golangci-lint (.golangci.yml)
make fmt | tidy | clean

-tags fts5 is mandatory: SQLite's FTS5 module is conditionally compiled inside mattn/go-sqlite3. Without the tag, rms_item_cache_fts creation fails at boot.

Docker uses a multi-stage build with Debian-slim runtime: both mattn/go-sqlite3 (CGO) and gopacket/pcap (libpcap) need shared libs, so distroless won't work. Mount ./data to persist the DB and emblems across containers.

8 · Package dependency direction who imports whom

The graph is intentionally one-way: leaf utilities never import the application core. internal/discord in particular accepts its parse + post functions as parameters rather than importing internal/server, which keeps the bot independently testable.

Solid arrows = static Go import. The single dashed arrow shows the one cross-leaf edge: internal/discord imports gemini.TradeResult for its callback signatures but does not import internal/server. internal/server imports every leaf package; cmd/server imports only config + server. web/ is embedded via //go:embed rather than imported as Go code.

9 · Testing & observability what's covered, what isn't

Pure-math & parsing

Covered: internal/xp, internal/config, internal/itemdb. These have unit tests because they are I/O-free.

Server-side

Limited: internal/server/handlers_test.go and template_parse_test.go verify template-parsing and a few handlers. Broader handler tests are gated on the Phase 10 repository-interface work in RESTRUCTURE_PLAN.md.

Logging

Standard log only — prefixed tags ([I] [Job] …, [E] [HTTP] …) make grep-by-component easy. There is no metrics / tracing pipeline; observability is "stdout + the admin page".
